## Model Correction Notes

### How To Record Lessons

- 经验用于纠正模型容易犯的判断错误，内容聚焦下一次决策方式。
- 经验应抽象成可迁移规则：何时先排查什么、何时提升抽象、何时收窄职责边界。
- 长期架构边界可以点名目录或模块；单次实现清单、字段流水账和临时修复细节应沉淀为判断原则。

### Debugging Third-Party Components

- AntD 组件表现异常时，优先检查本项目传入参数、主题 token、布局容器和组合方式。
- Table 出现滚动条颜色、背景、空白、错位、横向滚动或 fixed column 问题时，先核对列宽总和、`scroll.x`、固定列宽度、操作列实际内容和容器尺寸。
- 新增第三方组件样式覆盖前，先定位根因；样式覆盖适合处理第三方内部结构、token 联动和 className 覆盖能力不足的场景。

### Layout And Measurement

- 页面级布局先建立稳定高度链路、滚动容器和内容区域尺寸，再处理局部视觉细节。
- 表格、虚拟列表和看板泳道依赖确定高度；优先检查父容器 `min-height`、`flex`、`height: 100%` 和滚动区域边界。
- 测量型组件放进 Tabs、Collapse、Drawer 等隐藏容器时，优先判断隐藏状态下 ResizeObserver 或 DOM 测量得到 0 高度的影响。
- 虚拟列表优先使用成熟库承载滚动、测量、复用和边界处理；业务代码只提供数据、稳定 key、item 渲染和容器尺寸。
- 修改布局类问题后，验证数据少、数据多、横向滚动、fixed column、Tab 切换和分页切换场景。

### Abstraction And File Boundaries

- 可复用 wrapper 和 hook 的职责应围绕真实重复点设计；wrapper 负责测量和注入派生属性，业务配置由子组件声明。
- 已经能从子组件 props 合并得到的信息，优先从子组件读取并不可变合并，减少重复入参并保持调用方对象稳定。
- 对 AntD 这类复杂组件封装时，保留原生能力和原生样式，封装层只补齐项目布局约束。
- 单页面独有且结构稳定的配置优先就近放在页面组件内；表格 columns 强绑定单页交互时也保留在页面组件内。
- 少量固定枚举 UI 元素需要表达明确业务顺序时，直接显式写出；循环适合真实动态列表或重复结构收益明显的场景。
- `components` 目录只放 React 组件；格式化、筛选归一化、常量和类型工具放到页面级 `utils`、`constants` 或同层模块。

### Styling And Product Fit

- 普通 HTML 元素的布局、间距、颜色和状态样式优先使用 Tailwind；`antd-style` 用于 AntD 内部结构、token 联动和第三方组件覆盖。
- 去掉视觉效果时，同步收敛样式来源，让渐变、阴影和主题字段与当前设计意图一致。
- 管理端页面美化优先服务信息密度、清晰层级、稳定状态色和数字可读性；装饰性背景应服务信息分区。
- Apple 风格落到管理后台时，应体现为细边框、轻阴影、半透明层级、紧凑间距和清晰排版。

### State And Data Flow

- 分页这类跨页面一致的状态，优先抽成单个对象返回；默认配置放进 hook 内部，页面只传业务差异。
- 筛选条件变化和 pageSize 变化会影响分页边界，分页 hook 需要统一处理回到第一页。
- 接口一次返回完整候选池但页面仍使用表格分页时，分页应定义为纯前端展示状态，并明确筛选、刷新和 Tab 切换的页码重置规则。
- 状态到分组、泳道、阶段这类业务映射应建立单一数据源，再派生查询结构、筛选条件和统计结果。

### Routing And Page Structure

- 详情页、看板页、编辑页这类独立任务页面应使用独立路由渲染；列表切换页适合放在 tab 容器内。
- 独立页面组件应放在页面目录外层；`components` 目录保留给列表行、筛选区、标签、格式化展示等局部组件。
- 提升隐藏详情路由或新增可见页面时，同步确认 URL 层级、侧边栏高亮、菜单排序、`navKey`、面包屑策略和多语言标题。

### Documentation To Implementation

- 用户先要求完善文档、后要求实现时，把文档视为实现规格；实现前再次对照文档确认布局、接口、状态和验收点。
- 用户说“参考某页面”时，先判断被点名文件是外层壳还是承载核心交互的子组件。
- CRUD 和页面规格文档应覆盖可执行的布局、数据流、状态、筛选分页、表格列、表单校验、操作反馈和异常态。

### Verification And Reporting

- 开始修改前先把本次任务触发的项目经验提取成检查清单；改动中逐项对照，最终汇报已应用经验和未触发经验。
- 接手已有脏工作区时，先区分本次改动和既有改动；验证命令出现既有 warning 或沙盒端口限制时，分别报告环境问题和本次代码检查结果。
- 开发环境 mock 应覆盖主流程状态变化，让列表刷新、操作成功、操作后移出候选池这类交互可以在无后端时验证。
- 通用逻辑优先使用成熟库、框架内建能力或项目已有工具；日期、集合转换、对象分组、键值索引和常见浏览器 hook 优先使用 `date-fns`、`lodash-es`、`usehooks-ts` 等既有能力。
