# 经验维护规范

## 结构

- 运行时经验只维护在 `docs/experience/README.md`。
- 所有经验都来自已落地项目规范、用户纠偏或开发问题复盘。
- 所有命中的经验都需要遵守。

## 条目格式

```md
### EXP-AREA-000 标题

一到两句散文：第一句说清触发场景，第二句说做法。触发条件嵌在正文里，不单独设字段。

BAD: 一行反例（有明显对比时才写，没有则省略整行）
GOOD: 一行正例（有明显对比时才写，没有则省略整行）
```

## 写作规则

- 每条经验只解决一个稳定判断问题。
- 触发条件嵌在正文第一句，不单独设"触发"字段。
- 用短句写规则，少解释背景。
- 有明显对比时加 BAD/GOOD 示例行；没有对比就省略，不强制。
- BAD 尽量补一句后果（用 `——` 接），说清为什么错、边界在哪，比只给错误写法更好迁移。
- BAD/GOOD 必须是同一段代码或同一场景的错误版和正确版，不能各说各的。
- 纯代码类的对比放进 ```ts / ```tsx 代码块，用 `// BAD` `// GOOD` 注释区分；文字类对比用行内 `BAD:` `GOOD:`。
- 能扩展旧经验时扩展旧经验，不新建条目。
- 临时字段清单、一次性修复细节沉淀为可迁移规则。
- README 超过可快速通读的长度时，先合并重复经验，再新增条目。
